# 0001 - Модульний Моноліт (поточний вибір)

**Дата:** 26.10.2025
**Статус:** Прийнято

## 1. Контекст

Проєкт "Довідник дорожніх знаків" знаходиться на початковій стадії розробки. Команда невелика (1 особа). Система має обмежену бізнес-сферу: керування знаками та керування користувачами (Auth).

## 2. Рішення

Проєкт буде розроблятися як **Модульний Моноліт (Monolithic Architecture with internal modules)**.

## 3. Аргументація

* **Низька складність на старті:** Моноліт забезпечує просту розробку, деплоймент та налагодження (достатньо запустити один Flask-сервер).
* **Спільна БД:** Спрощує транзакції та запити.
* **Ефективність команди:** Невелика команда не потребує складності оркестрації мікросервісів.
* **Гнучкість:** Поточна структура (з репозиторіями та DDD-моделями) дозволяє легко **розбити** систему на мікросервіси в майбутньому (наприклад, винести `users` в окремий Auth-сервіс).

## 4. Альтернативи

* **Мікросервіси (Microservices):** Відхилено. Передчасне ускладнення інфраструктури, необхідність додавати API Gateway, Docker Compose та витрачати час на міжсервісну комунікацію, що не приносить негайної бізнес-цінності.

## 5. Наслідки

* **Позитивні:** Швидкий цикл розробки, один релізний артефакт, спрощений Continuous Integration.
* **Негативні:** Всі модулі залежать від одного середовища виконання (Python/Flask), що може ускладнити масштабування в далекому майбутньому.

## 6. Умови перегляду (коли перейдемо на мікросервіси)

Ми повернемося до питання розбиття, коли:
1.  Розмір кодової бази зросте настільки, що час збірки перевищить 5 хвилин.
2.  Нам знадобиться використовувати різні технології для окремих модулів (наприклад, Go для Auth-сервісу).
3.  Навантаження на модуль "Користувачі" (Auth) стане значно вищим, ніж на модуль "Знаки", і їх потрібно буде масштабувати незалежно.
